## Purpose

Document every convention for creating, reviewing, and applying Supabase database migrations
for Rig Tools — including table DDL, RLS setup, policies, indexes, and seed data. Covers both
local development (Supabase CLI + Docker) and remote production (Supabase hosted project).

## When to Use

Activate when the task contains: **migration**, **schema**, **table**, **column**, **DDL**, **RLS**,
**policy**, **alter table**, **create table**, **drop table**, **index**, **foreign key**, **seed**,
**rollback**, **supabase db**, **supabase migration**, **db push**, **db reset**, **db diff**.

## Conventions

1. **All schema changes go into migration files.** Never run DDL manually in the SQL editor for production.
2. **Migration files are immutable once pushed.** Never edit a file that has already been applied remotely. Create a new migration to fix it.
3. **One migration per logical change.** A table + its RLS + its policies + its indexes = one migration file.
4. **RLS is always enabled in the same migration that creates the table.** Never ship a table without RLS.
5. **Naming:** `{YYYYMMDDHHMMSS}_{description_snake_case}.sql` — generated by `supabase migration new`.
6. **File location:** `supabase/migrations/` — this directory is the source of truth for the schema.
7. **`supabase/seed.sql`** is for development/test data only — never production secrets or real user data.
8. **Destructive changes** (DROP COLUMN, DROP TABLE, type change) require a multi-step migration sequence: expand (add new) → backfill → cut over → contract (remove old). Never drop in a single migration unless the table was just created in development.
9. **`db/` layer functions** (`api/db/*.py`) must be updated whenever the underlying table schema changes — column renames, new non-nullable columns, removed columns.
10. **`supabase db diff`** is the preferred way to generate migrations from Supabase dashboard SQL changes — run it, review the output, then save as a named migration.

```
└── migrations/
    ├── 20240101000000_init.sql                   # Initial schema
    ├── 20240215120000_add_wells_table.sql
    ├── 20240301090000_add_rls_to_wells.sql       # Only if RLS was missed — prefer combined
    └── {YYYYMMDDHHMMSS}_{description}.sql        # Every future change
```

## CLI Reference

### Setup

```bash
# Install Supabase CLI (once, macOS/Linux)
brew install supabase/tap/supabase

# Link project to Supabase hosted instance
supabase login
supabase link --project-ref <project-ref>   # Found in dashboard URL: supabase.co/dashboard/project/<ref>

# Start local Supabase stack (Postgres + Auth + Storage + Studio)
supabase start
# → Local Studio:   http://localhost:54323
# → Local API:      http://localhost:54321
# → Local DB:       postgresql://postgres:postgres@localhost:54322/postgres
```

### Creating migrations

```bash
# Create a blank timestamped migration file
supabase migration new add_wells_table
# → Creates: supabase/migrations/20260315143000_add_wells_table.sql

# Generate migration from diff between local DB and current schema files
supabase db diff --schema public -f add_wells_table
# → Creates: supabase/migrations/20260315143000_add_wells_table.sql (auto-filled with diff SQL)
```

### Applying migrations

```bash
# Apply to local Supabase (reset + replay all migrations)
supabase db reset

# Push migrations to REMOTE (production) Supabase project
supabase db push

# Check migration status (which have been applied remotely)
supabase migration list
```

### Inspecting schema

```bash
# Pull current remote schema into a migration file
supabase db pull

# Dump current local schema (for review)
supabase db dump --schema public
```

## Migration File Template

```sql
-- supabase/migrations/{timestamp}_{table_name}.sql
-- Purpose: Create {table_name} table with RLS and access policies.
-- Author: migration-agent
-- Date: {date}

-- ── Table ─────────────────────────────────────────────────────────────────
CREATE TABLE IF NOT EXISTS public.{table_name} (
    id          uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     uuid        NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    created_at  timestamptz NOT NULL DEFAULT now(),
    updated_at  timestamptz NOT NULL DEFAULT now()
    -- Add feature columns here
);

-- ── Indexes ───────────────────────────────────────────────────────────────
CREATE INDEX IF NOT EXISTS {table_name}_user_id_idx ON public.{table_name} (user_id);

-- ── Updated-at trigger ────────────────────────────────────────────────────
CREATE OR REPLACE FUNCTION public.set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_{table_name}_updated_at
  BEFORE UPDATE ON public.{table_name}
  FOR EACH ROW EXECUTE FUNCTION public.set_updated_at();

-- ── Row-Level Security ────────────────────────────────────────────────────
ALTER TABLE public.{table_name} ENABLE ROW LEVEL SECURITY;

-- Users can only see their own rows
CREATE POLICY "{table_name}_select_own"
    ON public.{table_name}
    FOR SELECT
    USING (auth.uid() = user_id);

-- Users can only insert rows owned by themselves
CREATE POLICY "{table_name}_insert_own"
    ON public.{table_name}
    FOR INSERT
    WITH CHECK (auth.uid() = user_id);

-- Users can only update their own rows
CREATE POLICY "{table_name}_update_own"
    ON public.{table_name}
    FOR UPDATE
    USING (auth.uid() = user_id)
    WITH CHECK (auth.uid() = user_id);

-- Users can only delete their own rows
CREATE POLICY "{table_name}_delete_own"
    ON public.{table_name}
    FOR DELETE
    USING (auth.uid() = user_id);
```

## Common Migration Patterns

### Add a column (non-breaking)

```sql
-- supabase/migrations/{timestamp}_add_{column}_to_{table}.sql
ALTER TABLE public.{table_name}
    ADD COLUMN IF NOT EXISTS {column_name} {type} {constraints};
```

### Rename a column (multi-step — safe)

```sql
-- Step 1: Add new column, keep old one
ALTER TABLE public.{table_name}
    ADD COLUMN IF NOT EXISTS {new_column} {type};

-- Step 2: Backfill (in same migration if fast, or separate script for large tables)
UPDATE public.{table_name} SET {new_column} = {old_column};

-- Step 3: Add NOT NULL constraint after backfill
ALTER TABLE public.{table_name}
    ALTER COLUMN {new_column} SET NOT NULL;
```

```sql
-- Step 4: (separate later migration) Drop old column after code is deployed
ALTER TABLE public.{table_name}
    DROP COLUMN IF EXISTS {old_column};
```

### Add RLS policy to existing table

```sql
-- supabase/migrations/{timestamp}_add_rls_{table}.sql
ALTER TABLE public.{table_name} ENABLE ROW LEVEL SECURITY;

CREATE POLICY "{table_name}_select_own"
    ON public.{table_name}
    FOR SELECT
    USING (auth.uid() = user_id);
```

### Create an enum type

```sql
CREATE TYPE public.{enum_name} AS ENUM ('value_a', 'value_b', 'value_c');

-- Add values later (cannot remove values without recreating the type):
ALTER TYPE public.{enum_name} ADD VALUE IF NOT EXISTS 'value_d';
```

### Grant service-role schema access

```sql
-- Required when service role needs to bypass RLS for specific tables
GRANT ALL ON public.{table_name} TO service_role;
```

```bash
SUPABASE_PROJECT_REF=your-project-ref
```

## CI/CD Integration

```yaml
# .github/workflows/migrate.yml (example)
name: Apply Migrations

on:
  push:
    branches: [main]
    paths:
      - 'supabase/migrations/**'

jobs:
  migrate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: supabase/setup-cli@v1
        with:
          version: latest
      - run: supabase link --project-ref ${{ secrets.SUPABASE_PROJECT_REF }}
        env:
          SUPABASE_ACCESS_TOKEN: ${{ secrets.SUPABASE_ACCESS_TOKEN }}
      - run: supabase db push
```

## Anti-Patterns

- **Editing a pushed migration file** — already-applied SQL; it will not re-run. Create a new migration.
- **Running DDL in the Supabase SQL editor without saving to a migration file** — the change will be lost on `db reset` and not replicated to other environments.
- **Creating a table without enabling RLS** — any row is accessible to anonymous users via the API.
- **Using the service-role key from `api/db/` for user-facing queries that should be RLS-protected** — always use the anon key with the user's JWT.
- **Storing migration SQL in `api/db/` Python files** — migrations live in `supabase/migrations/` only.
- **Using `serial` or `integer` for primary keys** — always use `uuid DEFAULT gen_random_uuid()`.
- **Skipping the `IF NOT EXISTS` / `IF EXISTS` guards** — migrations must be idempotent for `db reset` safety.
- **Mutating enum values** — SQL enums cannot have values removed; design carefully or use a lookup table instead.

## Checklist

- [ ] Migration file placed in `supabase/migrations/`
- [ ] File named `{YYYYMMDDHHMMSS}_{description_snake_case}.sql`
- [ ] Table uses `uuid PRIMARY KEY DEFAULT gen_random_uuid()`
- [ ] `created_at` and `updated_at` columns present with defaults
- [ ] `updated_at` trigger created (or `moddatetime` extension used)
- [ ] `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` present
- [ ] RLS policies cover SELECT, INSERT, UPDATE, DELETE
- [ ] Policies use `auth.uid()` — never hardcoded user IDs
- [ ] All `CREATE` / `DROP` / `ALTER` statements use `IF NOT EXISTS` / `IF EXISTS`
- [ ] `api/db/{table}.py` updated to match new schema
- [ ] `api/models/{table}.py` Pydantic models updated if columns changed
- [ ] `supabase/seed.sql` updated if new tables need dev seed data
- [ ] Migration verified locally with `supabase db reset` before pushing
